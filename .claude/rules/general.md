# 一般ルール

## コーディングスタイル

- インデントはスペース2つ（プロジェクトの `.editorconfig` に従う）
- 変数名・関数名はキャメルケース、定数はスネークケース大文字
- コメントは「なぜ」を書く。「何をしているか」はコードから明らか

## 変更の方針

- 依頼された変更のみ行う。関係ないリファクタリングや「改善」は行わない
- 既存のコードスタイル・パターンに合わせる
- 新しい依存関係の追加は慎重に。追加が必要な場合は理由を述べる

## Git

- コミットメッセージは日本語 or 英語で簡潔に（プロジェクトに合わせる）
- ブランチ名: `feature/説明`, `fix/説明`, `chore/説明`

## Plan モードの活用

以下の場合は実装前に Plan モードに入り、方針をユーザーと合わせる：

- 変更が複数ファイルにまたがる（目安：3ファイル以上）
- 不可逆な操作（削除、DB スキーマ変更、大規模リファクタなど）
- 要件が曖昧で解釈が分かれそうな時

以下の場合は不要：

- 明確な1ファイルの変更（タイポ修正、変数名変更など）
- ユーザーがすでに方針を明示している時

**一文ルール**：変更内容を一文で説明できるなら不要。できないなら Plan モードに入る。

## サブエージェントの活用

以下の場合は積極的にサブエージェントを使う：

- 広範なコードベース探索（キーワード検索、ファイル特定など）
- 独立した複数タスクの同時実行
- メインのコンテキストを汚染しそうな大量の調査

使い分け：

- 探索・調査 → Explore エージェント
- 実装方針の設計 → Plan エージェント
- 独立タスクの並列化 → general-purpose を複数並列
